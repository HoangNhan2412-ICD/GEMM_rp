## 4. RTL rules

Use synthesizable Verilog/SystemVerilog.

Rules:
- Do not use `#delay` in synthesizable RTL.
- Testbenches may use delays, tasks, display, file I/O, and waveform dump.
- Reset all state registers.
- Avoid inferred latches.
- Use explicit bit widths.
- Carefully handle signed/unsigned arithmetic.
- For GEMM:
  - input data may be INT8 signed unless the existing RTL specifies otherwise
  - multiplication result should be at least signed 16-bit
  - MAC accumulation should be 32-bit or wider unless the design intentionally uses another width
- Do not change top-level interface without explaining why.
- If an interface change is necessary:
  - update `docs/interface_protocol.md`
  - update `docs/signal_table.md`
  - update `docs/change_log.md`
- Preserve existing module names unless there is a strong reason to rename.

## 5. Top module policy

Potential top modules:
- `MM_ultra`: preferred compute-core baseline if `GEMM_TOP` is too AXI/wrapper-dependent.
- `GEMM_TOP`: use only if dependencies and interface are clear enough.
- `PE` or `PE_array`: use for unit testing and smaller synthesis experiments.

For synthesis baseline:
- Prefer `MM_ultra` unless the Vivado script already proves another top works.
- Clearly state the synthesis top in:
  - `docs/synthesis_report.md`
  - `docs/resource_report.md`
  - `docs/timing_report.md`

For simulation baseline:
- Start with `PE`.
- Then test `PE_array`.
- Then test `MM_ultra`.
- Then test `GEMM_TOP` if interface and dependencies are clear.

## 6. Verification rules

Every important module should have a testbench.

Priority testbenches:
1. `tb_PE.v`
2. `tb_PE_array.v`
3. `tb_MM_ultra.v`
4. `tb_GEMM_TOP.v` if the wrapper interface is clear

Required testcase categories:
1. Reset behavior
2. Zero input
3. Positive values
4. Negative signed values if design is signed
5. Mixed sign values
6. Boundary values
7. Simple matrix multiplication known answer
8. Back-to-back start if interface supports it
9. Start/busy/done protocol
10. Output valid timing
11. Output stability after done
12. Width/overflow sanity check

Every testbench should:
- generate clock/reset
- print PASS/FAIL
- compare against expected output where possible
- dump waveform if supported
- document limitations

Never mark a test PASS unless the simulator actually ran.

## 10. DSP inference investigation

Because current parsed result shows `DSP = 0`, investigate before optimizing.

Create/update:

```text
docs/dsp_inference_plan.md
```

Questions to answer:
1. Which top module was synthesized?
2. Does that top include the PE/MAC compute path?
3. Are multipliers present in the elaborated/synthesized design?
4. Were multipliers optimized away due to constants, unconnected outputs, or inactive control?
5. Does the RTL coding style prevent DSP inference?
6. Are operand widths too small, causing Vivado to map multipliers to LUTs?
7. Should `(* use_dsp = "yes" *)` be tested?
8. Should PE/MAC be synthesized separately to check DSP inference?

Do not change the architecture aggressively until the cause is understood.

Allowed safe experiment:
- Create a separate branch and optional experimental PE synthesis top.
- Compare resource reports.
- Clearly mark results as experimental.
